* Engagement Reference Guide
:PROPERTIES:
:CATEGORY: Playbook
:END:

A practical reference for running client engagements using the full tool stack:
org-mode captures, org-roam, the Fractional CTO playbook, mattpocock/skills,
and gstack. Read top-to-bottom once to understand the system; use as a lookup
reference after that.

All engagement captures use a single =e= prefix (=C-c c e<key>=) and route to the
*active client*. Set it once per session with =M-x org-fractional-cto-set-active-client= and every
capture files to that client automatically — no per-client prefix letters, no
copy-pasted config. Switch clients mid-session by setting a different active client.

The mechanics (captures, dashboard, onboarding) are provided by the
=org-fractional-cto= package ([[file:../README.org][README]]). This guide is the *methodology* layer
on top of it: when to use each capture, the operating rhythm, and the hats.

** Legend

*** TODO keywords
=org-fractional-cto-setup= registers these globally (via =org-todo-keywords=), so
they work in every buffer without touching your personal Org config; client hubs
also declare them locally with =#+TODO: TODO NEXT INPROGRESS WAITING | DONE
CANCELLED=. Customize via =org-fractional-cto-todo-keywords=.

| Keyword       | Meaning                                                   |
|---------------+-----------------------------------------------------------|
| =TODO=        | Open action you own                                       |
| =NEXT=        | The next physical action (GTD-style)                      |
| =INPROGRESS=  | Actively being worked                                     |
| =WAITING=     | *Delegated* — owned by someone else, you're tracking it   |
| =DONE=        | Completed                                                 |
| =CANCELLED=   | Dropped                                                   |

=WAITING= is load-bearing: it separates "things I owe" from "things I'm owed,"
and feeds the dashboard's *Delegated* block.

*** Engagement stages (tag on the engagement heading)
One stage tag sits on the top =* … Engagement= heading and is inherited by the
whole tree. Switch it with =M-x org-fractional-cto-set-stage=.

| Stage       | Meaning                                          |
|-------------+--------------------------------------------------|
| =LEAD=      | Captured from a pre-sales call; raw              |
| =QUALIFIED= | Researched and worth pursuing (pre-contract)     |
| =ACTIVE=    | Won / engaged — the default for new-client       |
| =LOST=      | Did not convert                                  |
| =DORMANT=   | Paused / on hold                                 |

The pipeline view (=C-c a P=) lists every =LEAD=/=QUALIFIED= engagement across
all clients, one line per prospect.

| =[#B]= | Important, not urgent                            |

** File Organization

The package ships the templates and these guides; your engagements live in
=org-fractional-cto-clients-directory= (default =<org-directory>/clients=).

The package (shareable, read-only to you):

#+begin_example

A non-client inbox (=refile.org= or your =org-default-notes-file=) still holds
generic captures (=t=, =n=, =m=); engagement captures bypass it entirely.

*** The capture flow

Client captures go *directly* to the relevant section of =<client>.org= via
=file+headline= — no inbox step, no refiling needed. Open =<client>.org= (or any
client file) to see everything that has been captured for that account, organised
by section.

=refile.org= is still the inbox for *non-client* captures (generic todos, notes,
meeting entries via the =t=, =n=, =m= keys). The discipline of clearing it still
applies, but client work bypasses it entirely.

*** The lookup flow

1. *Agenda dashboard* — =C-c a E= opens the active client's dashboard: 7-day agenda,
   delegated items awaiting response, blockers & escalations, open actions,
   commitments with deadlines, active risks. This is the daily starting point.
   (If no active client is set, it prompts you to pick one.)
2. *Browse by section* — =C-x C-f <client>.org= then navigate headings (=C-c C-n/p=
   to move between headings, =S-TAB= to collapse/expand). Each section contains
   all captures of that type in reverse-chronological order.
3. *Tag search* — =C-c a m TAGNAME= to find entries across all agenda files by tag
   (e.g. =RISK=, =COMMITMENT=, =TECHDEBT=).
4. *Full-text search* — =C-c a s= to search across all agenda files by keyword.
5. *org-roam* — =M-x org-roam-node-find= for the longer-form reference docs
   (role plan, system discovery).

** Capture Templates — Complete Reference

All engagement templates use the pattern: =C-c c e<key>=. Captures file *directly*
to the relevant section of the active client's =<client>.org= (set the active
client with =M-x org-fractional-cto-set-active-client= to avoid being prompted each time).
Clocked entries start the org-clock automatically.

*** Relationship & Communication Templates

**** =es= — Standup Notes
/Use:/ Attending any team standup. One capture per session you attend.
/Clock:/ Yes — tracks time spent in standups.
/Fields:/ Date (auto), client name.
/Structure:/ Per-team sections (Shipped / Next / Blockers) + Actions/Escalations.
/After capturing:/ Scan for any blocker that needs a risk entry (=er=) or
commitment capture (=ec=). Refile to client's discovery or ops folder.

**** =em= — Client Meeting
/Use:/ Any call or meeting with the client — planning, status, ad hoc.
/Clock:/ Yes.
/Fields:/ Meeting title, client attendees, our attendees.
/Structure:/ Agenda → Discussion → Decisions → Commitments (both sides) → Actions → Follow-up deadline.
/After capturing:/ Immediately capture any commitments (=ec=). Send written
summary to client within 2 hours. Never let a meeting end without a follow-up
summary — this is how trust is built.

**** =ei= — Internal Sync
/Use:/ Weekly internal sync with team leads — always before client-facing comms.
/Clock:/ Yes.
/Fields:/ Date, client name.
/Structure:/ Commitments vs. reality gap | Blockers to escalate | Risk updates |
Client comms review | Actions.
/Rule:/ Nothing surfaces to the client that hasn't passed through this sync first.

**** =ep= — Stakeholder Profile
/Use:/ First time you meet or learn about a significant client contact. One profile
per person. Update it as the relationship develops.
/Clock:/ Yes.
/Fields:/ Full name, role, organisation.
/Key sections:/ Their personal definition of success (not the project's) | Key
priorities & concerns | Communication preferences | Relationship status |
How to engage them effectively.
/Why it matters:/ Stakeholder maps decay fast. Documenting what makes each person
tick lets you tailor every communication.

**** =ec= — Commitment Captured
/Use:/ Any time a commitment is made — by you *or* by the client — in any meeting.
Capture it immediately, while still in the session.
/Clock:/ Yes.
/Fields:/ Commitment description, due date (becomes a =DEADLINE=), owner.
/After capturing:/ Refile to the commitment log in the client's document. Review
all open commitments at each internal sync.

**** =eh= — Client Health Check
/Use:/ Monthly. Three pulse questions sent to the client (or discussed in a call).
/Fields:/ Month, year.
/Questions (built in):/ What's working well? What would you change? What would you
love to see in the next 30 days?
/After capturing:/ Act on one specific thing from their feedback before next month.
The pattern of not acting on feedback is invisible until it's too late.

**** =eq= — QBR (Quarterly Business Review)
/Use:/ Quarterly formal checkpoint with client stakeholders.
/Clock:/ Yes.
/Fields:/ Quarter, client attendees, our attendees.
/Structure:/ Deliverables table | Business impact | Technology roadmap review |
Strategic recommendation | Next quarter priorities | Renewal/expansion | Decisions
needed | Action items.
/After capturing:/ Send written summary within 24 hours. The written summary is
part of the ceremony — don't skip it.
/Strategic recommendation:/ Always include one recommendation they didn't ask for.
Frame it as business opportunity first, technology second. This is what separates
a vendor from a strategic partner.

*** Action Tracking & Delegation Templates

**** =ew= — Action Item (quick TODO)
/Use:/ A task you're taking on yourself. The fast path for ad-hoc work.
/Clock:/ Yes.
/Fields:/ Action description, due date (becomes a =DEADLINE=).
/After capturing:/ Shows up in =C-c a E= → "Open Actions". Re-prioritise as needed.

**** =eg= — Delegate Action (WAITING)
/Use:/ Any time you hand a task to a team member. This is the core delegation tool.
/Fields:/ Task, assignee (=ASSIGNED_TO= property), expected delivery (=DEADLINE=),
your check-in date (=SCHEDULED=), success criteria.
/Why WAITING:/ The entry is created in the =WAITING= state, not =TODO=. This separates
"things I owe" from "things I'm owed" — they surface in the dedicated
"Delegated — Awaiting Response" block of =C-c a E=.
/After capturing:/ The =SCHEDULED= date is your follow-up trigger. If the check-in
passes with no update, escalate. Review every WAITING item at the Friday weekly review.
/Discipline:/ Delegation without a check-in date is hoping, not managing. Always set one.

**** =eb= — Blocker / Escalation
/Use:/ The moment something is blocking delivery and you can't clear it yourself.
/Fields:/ What is blocked, what feature/stream it blocks (=BLOCKING=), who can remove
it (=UNBLOCK_OWNER=), root cause, options, escalation path, resolve-by deadline.
/Priority:/ Created as =[#A] TODO=. Blockers are always top priority — they compound.
/After capturing:/ Surfaces in =C-c a E= → "Blockers & Escalations". If unresolved by
the deadline, escalate via the named path (standup → internal sync → client).

**** Acting on an item that already exists (=delegate-at-point=, =block-at-point=)
=ew=, =eg= and =eb= /create/ entries. Two interactive commands act on the Org
heading already at point, so you never retype work that is already captured.

- =M-x org-fractional-cto-delegate-at-point= (=g= in the command map) — the =eg=
  gesture applied in place. Sets the heading to =WAITING=, adds the =DELEGATED=
  tag, records =ASSIGNED_TO= and =DELEGATED_ON=, and =SCHEDULED=s your check-in;
  optionally sets a delivery =DEADLINE=. The item moves to the dashboard's
  "Delegated — Awaiting Response" block. An assignee is required.
- =M-x org-fractional-cto-block-at-point= (=b= in the command map) — the =eb=
  gesture applied in place. Files a =[#A]= blocker into the client's Blockers
  section with =BLOCKING= linked back to the action, and inserts a
  back-reference under the action. Optional resolve-by =DEADLINE=.

/Discipline:/ reach for these when the work is already on a heading; reach for
=eg= / =eb= when you are capturing something new. The back-link is an internal
=[[*heading]]= link, so keep action headings distinctly named.

**** =eW= — Weekly Review
/Use:/ Friday, 15 minutes, before the week closes. Your delegation and team pulse review.
/Clock:/ Yes.
/Fields:/ Week ending.
/Structure:/ Shipped this week | What slipped and why | Delegation inbox check-in
table | Blockers to escalate | Key decisions | Team pulse | Top 3 next week | Comms to send.
/The discipline:/ This is where you reconcile every WAITING item against reality.
Anyone who's gone quiet gets a direct ping — don't wait for the next standup.
/After capturing:/ Each "change/action needed" becomes a TODO. Use =C-c a E= delegated
block as the source of truth while filling the inbox table.

**** =eP= — Person / Team Member Note
/Use:/ Capturing something about a team member — capacity, a concern, a win, a
development need. Builds the People section over time.
/Fields:/ Name, role/stream, =STREAM= property.
/Why it matters:/ As a fractional CTO you're managing people you didn't hire and
don't see daily. Tracking capacity signals (who's overloaded, who's a single point
of failure) is how you spot delivery risk before it becomes a missed deadline.
/Maintain:/ The People section also holds the Team Roster and Stream → Lead map —
keep those current as the team changes.

*** Risk & Delivery Templates

**** =er= — Risk Item
/Use:/ Any new risk to the engagement or delivery — technical or otherwise.
Capture immediately when identified; don't wait for the next sync.
/Fields:/ Description, likelihood (High/Medium/Low), impact (High/Medium/Low), owner, mitigation.
/After capturing:/ Add to the risk register in the master discovery doc. Review
at every internal sync. A risk that sits in refile.org is not managed.

**** =ee= — Scope Change / Out-of-Scope Flag
/Use:/ Any time you identify work that is outside the current SOW, or the client
asks for something that may be out of scope.
/Fields:/ Description, who identified it, SOW status (out/in/grey area).
/Rule:/ Never agree to out-of-scope work in a client meeting. Capture it here,
bring it through the internal sync, then respond to the client with a clear framing.
/After capturing:/ Internal sync review → commercial assessment → client response.

**** =ef= — Post-Mortem / Incident
/Use:/ Any production incident, delivery failure, or significant quality issue.
/Fields:/ Incident title, date, severity, duration, what was affected.
/Structure:/ What happened | Root cause | How we fixed it | Was client notified? |
Prevention | Actions.
/Timing:/ Capture within 24 hours. The detail fades quickly.
/Client communication:/ "Here's what happened, here's what we did, here's what
we're doing to prevent recurrence." In that order.

**** =eR= — Retrospective
/Use:/ End of each sprint or delivery cycle. Distinct from a post-mortem — a
retro is *proactive and routine* (team process), a post-mortem is *reactive* (a
failure). Run a retro whether or not anything went wrong.
/Clock:/ Yes.
/Fields:/ Sprint/period, facilitator, attendees.
/Structure:/ Sprint goals vs. actuals (table) | What went well | What didn't go
well | What we'll change (owned, time-boxed) | Team health signals | Action items.
/Pair with:/ gstack =/retro= as the facilitation prompt before or during the session.
/After capturing:/ Each "what we'll change" row becomes a TODO in the Actions section.
/Discipline:/ Focus on the system, not the person. "What process let this happen?"
not "who dropped this?" The retro is for improving how the team works, not assigning blame.

*** Technical Templates

**** =ed= — Discovery Session
/Use:/ Structured conversation with any team lead to map their stream. One per stream.
/Clock:/ Yes — these are significant sessions.
/Fields:/ Team/role, client, attendees, focus area, stream name.
/Structure:/ Team context | Active workstreams (table) | Tech stack (table) |
Architecture & integrations | Pain points | Quality & testing | Deployment & ops |
Tech debt (table) | Security observations | Innovation opportunities | Open questions |
Follow-up actions.
/Before the session:/ Run =/grill-me= (mattpocock/skills) to surface your assumptions.
/After the session:/ Refile immediately. Update cross-cutting sections of the master
discovery doc. Run a synthesis pass after every 2–3 sessions.

**** =ea= — Architecture Decision Record (ADR)
/Use:/ Any significant technology or architecture decision — things that are hard to
reverse, affect multiple streams, or have long-term cost implications.
/Clock:/ Yes.
/Fields:/ Decision title, client, status (Proposed/Accepted/Deprecated/Superseded).
/Structure:/ Context (the forcing situation) | Decision (what and by whom) |
Alternatives considered | Trade-offs accepted | Consequences (positive/negative/risks) |
Review date.
/Rule:/ If you're spending more than 30 minutes deciding something, it warrants an ADR.
A decision not documented is a decision that will be re-litigated.

**** =eA= — Architecture Health Review
/Use:/ Quarterly. A structured assessment of the full system's health.
/Clock:/ Yes.
/Fields:/ Quarter, year, client, your name, teams reviewed.
/Structure:/ Scalability | Maintainability | Integration fragility | Dependency risk |
Technical debt table | Technology roadmap review | Recommendations | How to present.
/Framing for the client:/ "Transparency and forward planning" — never a problem list.
The goal is to demonstrate that you're ahead of issues, not behind them.

**** =ev= — Vendor / Tool Evaluation
/Use:/ Assessing any new tool, platform, or technology vendor.
/Clock:/ Yes.
/Fields:/ Vendor name, client, category, evaluator.
/Structure:/ Problem being solved | Recommendation (with rationale) | Alternatives |
Assessment (solves the problem? build vs. buy? lock-in? data handling? 2-year TCO) |
Decision deadline.
/Decision framework:/ Commodity infrastructure → buy. Differentiating capability → build.
Anything that touches customer PII → scrutinise the vendor's data handling terms first.

**** =et= — Technical Debt Item
/Use:/ Immediately when you identify tech debt — during discovery, code review,
architecture sessions, or standups.
/Fields:/ Description, area (Frontend/Backend/Infra/Integration/Data/Security),
severity (Critical/High/Medium/Low), discovery context, impact if unaddressed.
/After capturing:/ Add to the tech debt register in the master discovery doc.
Prioritise at the next architecture session. Debt that isn't visible isn't managed.

**** =ex= — Security Finding
/Use:/ Any security or compliance observation — immediately when found.
/Fields:/ Finding description, severity, area (PCI/GDPR/API/Auth/Data/Infra),
current status (Open/Mitigated/Resolved/Accepted), recommended action, owner.
Closed-out findings (Resolved/Mitigated) drop off the dashboard's "Open security
findings" block — the same rule as the risk register.
/Rule:/ Critical findings surface to the internal sync before end of day.
High findings surface within 48 hours. Never downgrade severity to avoid a
difficult conversation.

**** =en= — Innovation Pipeline Idea
/Use:/ Any time you spot something that could be done better, smarter, or that the
client doesn't yet know is possible. Capture without filtering — evaluate later.
/Fields:/ Title, category (AI/ML / Data / Platform / Integration / Music Tech / Other).
/Structure:/ The opportunity (business framing first) | The technology | Why now /
why this client | Rough effort | Next step.
/Pipeline review:/ Monthly — select one idea to develop into a brief. Present the
best one at each QBR as the "Strategic Recommendation."

** The Hat Model in Practice

Switching hats mid-session dilutes both. When you sit down to work, know which
hat you're wearing and stick to it for the duration of that block.

*** Before any significant session

Run one of these first — it takes 5 minutes and prevents hours of rework:

| Session type                                  | Run first                | Why                                                                       |
|-----------------------------------------------+--------------------------+---------------------------------------------------------------------------|
| Discovery interview                           | =/grill-me=              | Surface what you're assuming; don't anchor the conversation to your frame |
| Architecture decision                         | =/office-hours=          | Challenge your own framing before committing                              |
| Client meeting (complex topic)                | =/grill-with-docs= + SOW | Verify you're solving the right problem                                   |
| Starting implementation in an unfamiliar area | =/zoom-out=              | Map the system before touching it                                         |
| Handoff between sessions                      | =/handoff=               | Compact the context so the next session starts fresh                      |
| Security review                               | =/cso=                   | OWASP Top 10 + STRIDE, with false-positive filtering                      |

*** Hat descriptions

*Discovery hat*
You are listening, not advising. Your job is to surface reality, not impose
structure. Ask "what would break first if..." and "what's the thing nobody
talks about but everyone knows is a problem?" Take the full discovery template
notes. Don't jump to solutions in the session.

*Fractional CTO hat*
You are making or advising decisions. Use =/office-hours= to challenge your frame
first. Document every significant decision as an ADR. The value is not the answer —
it's the documented reasoning that lets the team build on it without re-litigating.

*Engagement Director hat*
You are managing the relationship. Speak business outcomes, not tech. Translate
everything into: what did the client gain? What risk did we reduce? What opportunity
does this create? Never bring a problem without a mitigation framing.

*QA Auditor hat*
Find the gaps the team has normalised. Ask specifically about: billing logic test
coverage, auth flow integration tests, regression coverage before releases. Teams
stop seeing their own coverage gaps after a while.

*Security Reviewer hat*
Assume breach, work backwards. What data would be exposed? What's the blast radius?
Use =/cso= (gstack) for the systematic scan. Flag immediately — severity doesn't
get negotiated.

*DevOps hat*
Optimise for recovery speed, not uptime. Questions: how fast can you detect an
incident? How fast can you roll back? What's the MTTR? Infrastructure-as-code
posture? Where are secrets stored?

*** How AI skills associate with an Org work item

There is no automatic binding between an Org heading and a skill — the link is
deliberate and manual, and that is the point. The flow is always:

1. *Pick the work item in Org* — an ADR you're about to write (=ea=), a tech
   spike (=ek=), a discovery session (=ed=), a security review (=ex=).
2. *Open the client's code repo* in your AI tool (Claude Code, etc.) and load
   its =CONTEXT.md= so the agent has the domain vocabulary.
3. *Run the matching skill* in that repo's AI session (tables below).
4. *Capture the outcome* back into Org with the relevant =e= key.

So Org tells you /what/ and /why/; the skill helps you /do/ it; Org records the
/outcome/. The skills live in your code repositories, never in =org-files=.

** Scenario Guides

*** "I just had a discovery session"
1. =C-c C-x C-o= — clock out of the session
2. Immediately refile the capture: =C-c C-w= → client's discovery section
3. Scan the session notes for:
   - Risks → =er= for each
   - Tech debt items → =et= for each
   - Security observations → =ex= for each
   - Innovation opportunities → =en= for each
4. Open the master system discovery doc, update:
   - Discovery Status table (tick the stream, add date)
   - The relevant per-domain deep dive section
5. If this was session 2 or 3: do a synthesis pass on cross-cutting sections
   (tech debt aggregate, integration map, security table)

*** "Something went wrong in production"
1. =ef= — post-mortem capture, clock in immediately
2. Fill in: what happened, severity, what is affected
3. Assess: does this need client notification? (Critical/High severity: yes)
4. If client notification needed: verbal first, written summary within 1 hour
   - Frame: "here's what happened / here's what we did / here's what prevents recurrence"
5. Internal: what process change prevents recurrence?
6. After resolution: complete the post-mortem entry (root cause, prevention, actions)
7. If it revealed a structural issue: =ex= (security) or =et= (tech debt)

*** "Client asked for something that might be out of scope"
1. In the meeting: "Let me come back to you on what that would look like" — never agree in the room
2. =ee= — scope change capture immediately after the call
3. Bring it to Monday's internal sync (=ei=)
4. Internal assessment: in scope / out of scope / grey area? Commercial impact?
5. Client response: frame as opportunity, not dispute
   - "Here's what that would involve, here's what it would enable, here's how we'd approach it"

*** "I want to pitch an innovation idea"
1. =en= — capture the idea immediately, don't filter it yet
2. Monthly pipeline review: shortlist to 1–2 ideas worth developing
3. Develop into a brief: business opportunity → technology → why now / why this client → rough effort
4. Run =/office-hours= (gstack) to stress-test the framing before presenting
5. Present at the next QBR as "Strategic Recommendation"
6. Frame: "here's what becomes possible if..." — never "here's what we could build"

*** "I need to make an architecture decision"
1. Run =/office-hours= (gstack) — challenge your own framing first
2. =ea= — ADR capture, clock in
3. Fill: Context (the forcing situation) → Alternatives considered → Trade-offs → Decision
4. For any vendor/tool involved: =ev= vendor evaluation
5. For security implications: =ex= security finding
6. Mark the ADR status: Proposed → get relevant leads to review → Accepted
7. Note review date — architecture decisions need scheduled re-evaluation

*** "Preparing for a QBR"
1. Three weeks before: run =/zoom-out= across the discovery doc — what's the narrative?
2. Two weeks before: run =/office-hours= — what's the one strategic recommendation?
3. One week before: draft QBR structure, translate all delivery into business outcomes
   - Not "we shipped X" — "X now means the client can do Y, which reduces risk Z"
4. Run =/grill-with-docs= on the QBR draft — where are you weak?
5. QBR session: =eq= — clock in
6. Within 24 hours: send written summary (non-negotiable)

*** "Starting a new engagement for the first time"
Follow the full [[*New Client Setup]] checklist above, then run Phase 0 of the
[[file:playbook.org][Fractional CTO Playbook]].

The single most important thing in the first week: run =/grill-me= (mattpocock/skills)
before your first substantive client conversation. You will have assumptions you
don't know you're making. Surface them before they cost you.
