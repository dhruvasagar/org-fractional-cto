#+title: org-fractional-cto User Guide
#+subtitle: A step-by-step walkthrough, from install to running an engagement
#+options: toc:2 num:nil
#+startup: overview

This guide walks you through *every feature*, in the order you would actually
meet them: install and verify, onboard a client, capture work, read the
dashboard, look across clients, and tailor the system. It assumes you are
comfortable in Emacs and Org (=M-x=, =C-c c= capture, the agenda dispatcher,
editing your init file).

It complements the other two documents rather than repeating them:

- This *guide* is the hand-held tour — /how do I use the tool?/
- [[file:playbook.org][playbook.org]] is the methodology — /how do I run the engagement?/
- [[file:reference.org][reference.org]] is the lookup — /what does each capture and tag mean?/

Work through it once, top to bottom, with Emacs open. Every step says what you
should *see* so you can confirm it worked before moving on.

* Before you start

You need:

- *Emacs 27.1+* and *Org 9.4+* (=M-x org-version=).
- =org-capture= bound to a key. This guide assumes the conventional
  ~C-c c~. If you have not bound it: ~(global-set-key (kbd "C-c c") #'org-capture)~.
- =org-agenda= bound to ~C-c a~ (the Org default).
- A directory to hold clients. The default is =<org-directory>/clients=; this
  guide uses =~/org/clients=.

Throughout, the example client is *Acme Corp* with the slug =acme=.

* Step 1 — Install and verify

** Install

With =use-package= from a local clone:

#+begin_src emacs-lisp
(use-package org-fractional-cto
  :load-path "~/src/org-fractional-cto"
  :after org
  :custom
  (org-fractional-cto-clients-directory "~/org/clients")
  (org-fractional-cto-author "Your Name")
  :config
  (org-fractional-cto-setup))
#+end_src

Or manually:

#+begin_src emacs-lisp
(add-to-list 'load-path "~/src/org-fractional-cto")
(require 'org-fractional-cto)
(setq org-fractional-cto-clients-directory "~/org/clients")
(org-fractional-cto-setup)
#+end_src

=org-fractional-cto-setup= does the following on every run:

- Installs the =e= capture group.
- Registers the dashboard (=E=) and prospect pipeline (=P=) agenda commands.
- Adds your existing client directories to =org-agenda-files=.
- Registers the workflow's TODO keywords (=TODO NEXT INPROGRESS WAITING …=)
  into the global =org-todo-keywords= and their keyword faces.
- Enables agenda tag inheritance (=org-agenda-use-tag-inheritance=) so the
  inherited client =#+filetags= tag is filterable.
- Wires the at-point action keybindings (Evil =, g= / =, b= in agendas and the
  optional non-Evil =org-fractional-cto-agenda-keymap-prefix=).

It is *idempotent* — safe to run again any time (e.g. after you edit
=org-fractional-cto-dashboard-blocks=). Re-running refreshes the captures and
the dashboard in place rather than duplicating them, so ~M-x
org-fractional-cto-setup~ is the way to apply config changes without restarting
Emacs.

** Verify

Four quick checks before you create anything:

1. *Setup ran.* ~M-x org-fractional-cto-setup~ returns without error.
2. *Dashboard is registered.* Press ~C-c a~. The dispatcher should list:
   =E   Fractional CTO — client dashboard=. (Press =q= to dismiss.)
3. *Captures are installed.* Press ~C-c c~. You should see
   =e   Engagement (select client)= in the menu. (Press =C-g= to dismiss.)
4. *Pipeline is registered.* Press ~C-c a~. The dispatcher should also list:
   =P   Fractional CTO — prospect pipeline=. (Press =q= to dismiss.)

If all four pass, the package is live.

* Step 2 — The two concepts you will lean on

Before onboarding, understand the two ideas the whole workflow rests on.

** The active client

Almost everything is scoped to one client at a time. Instead of a key prefix
per client, you set an *active client* once per session and every capture and
the dashboard follow it.

| Command                                      | Effect                                |
|----------------------------------------------+---------------------------------------|
| =M-x org-fractional-cto-set-active-client=   | Pick the client for this session      |
| =M-x org-fractional-cto-clear-active-client= | Go back to being prompted per capture |
| =M-x org-fractional-cto-switch-client=       | Set active *and* open its dashboard   |

With an active client set, ~C-c c e ...~ and ~C-c a E~ never prompt. With none
set, they prompt you to choose — nothing breaks, it is just one extra step.

** The client hub

Each client owns a directory with its hub, its context file, and a =templates/= folder:

#+begin_example
~/org/clients/
  acme/
    acme.org      ← operational hub: one section per capture type
    CONTEXT.md    ← domain glossary, people, tech stack (load in AI sessions)
    templates/    ← per-client copies of every capture template (override-able)
#+end_example

Onboarding copies the full bundled set of capture templates into =templates/=,
so every template is editable per client (see [[*Step 11 — Make it yours][Step 11]]).

The hub's section headings, the capture targets, and the dashboard's queries
are all generated from a single section list, so they agree by construction —
you never wire a capture to a heading by hand.

* Step 3 — Onboard your first client

Run:

: M-x org-fractional-cto-new-client

You are asked for two things:

1. *Client name (display)* — e.g. =Acme Corp=. Becomes the file =#+title=, the
   top-level heading, and the agenda =:CATEGORY:=. (The =#+AUTHOR= line is set
   separately from =org-fractional-cto-author=.)
2. *Client slug* — e.g. =acme=. Lowercase, no spaces; it names the directory
   and files, and is upper-cased into the Org tag (=acme= → =:ACME:=;
   non-alphanumerics become =_=, so =acme-eu= → =:ACME_EU:=).

The command then creates =acme/acme.org= and =acme/CONTEXT.md=, copies the
bundled capture templates into =acme/templates/=, adds the new directory to
=org-agenda-files= immediately (no restart), and opens =CONTEXT.md= for you to
fill in.

** Walk the hub it generated

Open =acme/acme.org=. At the top is a =#+TODO= line defining the workflow's
states:

#+begin_example
#+TODO: TODO NEXT INPROGRESS WAITING | DONE CANCELLED
#+end_example

(Note =INPROGRESS= has no hyphen — that is deliberate; see [[*Troubleshooting][Troubleshooting]].)
This line is a convenience, not a requirement: =org-fractional-cto-setup= already
registers the same keyword sequence into the global =org-todo-keywords=, so the
states work in any buffer even before you open a hub. The per-file line just
means a hub still cycles correctly on a machine where setup never ran.

Below the top-level =* Acme Corp Engagement  :ACTIVE:= heading is one =**= section
per capture type — =Actions=, =Delegations=, =Blockers=, =People=, =Commitments=,
=Risks=, =Meeting Notes=, …, =Retrospectives= — each tagged with only its type
subtag (e.g. =** Delegations  :DELEGATED:=). The client identity is declared once
at file level via =#+filetags: :ACME:= and is inherited by every heading — so
captures never carry a redundant per-item client tag. You do not edit these by
hand; captures file into them. Seeing the whole engagement in one outline is the
point — fold with =overview=, expand the section you need.

* Step 3b — Capture a prospect (pre-sales)

A prospect is just a client at an earlier stage. Before an engagement is won you
still want to capture calls, do research, and track follow-ups — and the
dashboard already does that. Onboard a prospect with:

: M-x org-fractional-cto-new-prospect

It scaffolds the same hub as a client but stamps the engagement heading with the
=LEAD= stage tag, sets the prospect active, and opens the *pre-sales call*
capture (=el=) so you record the conversation immediately. From there:

- =C-c c el= — capture each pre-sales call (the ask, pain, signals, next step).
- =C-c c eo= — log research findings as you learn about the company, market, and
  tech. Fold the durable facts into =CONTEXT.md=.
- =C-c c eF= — record a fit / qualification verdict (Pursue / Hold / Pass).
- =C-c c ew= / =eg= / =eb= — ordinary actions, delegations, and blockers work
  here too, so your pre-sales follow-ups land on the dashboard like any work.

As the prospect advances, move the stage with =M-x org-fractional-cto-set-stage=
(=LEAD= → =QUALIFIED= → =ACTIVE= when you win, =LOST= to close it out, or
=DORMANT= to park it without losing the record). Across all clients, =C-c a P=
shows the whole pipeline — every =LEAD= / =QUALIFIED= engagement, one line each.

If you have hubs created before stages existed, run =M-x
org-fractional-cto-upgrade-hub= on each. It: adds a =#+filetags: :SLUG:= line
(the client tag at file level), strips any now-redundant per-heading client tag
from every heading, appends any missing sections from the current section list,
ensures the engagement heading carries a stage tag (defaulting to =ACTIVE=), and
copies any bundled capture templates the client's =templates/= directory is
missing (templates you have already edited are left untouched). The command is
idempotent — safe to run more than once.

* Step 4 — Fill in CONTEXT.md

*Note:* This step describes the =new-client= path, where =CONTEXT.md= opens
automatically after onboarding. If you used =new-prospect= (Step 3b) instead,
the pre-sales call capture opened first; fill =CONTEXT.md= at your earliest
convenience (find it at =acme/CONTEXT.md=) and then skip to Step 6 — the
prospect is already set active.

=CONTEXT.md= is the human (and AI) brief for the engagement. It opens
automatically after =new-client=, pre-populated with section headers:

- *What We're Building* — one paragraph on the product and the mandate.
- *Domain Vocabulary* — the terms the team uses; the highest-leverage section.
- *Tech Stack* — as known at onboarding.
- *Key People* — client side and your side.
- *Active Work Streams* and *Current Priorities*.

Fill at least vocabulary, people, and priorities now; the rest accretes. If you
do hands-on implementation with an AI agent, load this file at the start of each
session so the agent speaks the client's language. It is plain Markdown so it
travels into any tool.

* Step 5 — Set the active client

*Note:* This step applies to the =new-client= path. If you used =new-prospect=
(Step 3b), the client was set active automatically — you can skip this step.

Back from =CONTEXT.md=:

: M-x org-fractional-cto-set-active-client

Choose =acme=. The echo area confirms =Active client: acme=. From here, every
capture and the dashboard target Acme with no prompting.

- =org-fractional-cto-clear-active-client= returns to per-capture prompting.
- =org-fractional-cto-switch-client= sets a *different* client active and opens
  its dashboard in one step — the fast path when you change context.

The active client is per-session state; it resets when Emacs restarts. Set it
once when you sit down to work on a client.

* Step 6 — Your first capture, in detail

Capture an action. Press ~C-c c e w~:

1. ~C-c c~ opens org-capture; =e= selects the engagement group; =w= selects
   *Action item*.
2. You are prompted for the action text, then a deadline.
3. Org files a =TODO= under Acme's =Actions= section, stamped with the creation
   time, and *clocks in* (the =ew= template sets =:clock-in=). The client tag
   =:ACME:= is inherited from =#+filetags= — it does not appear on the item itself.
   Because an active client is set, you are routed straight to Acme's hub — no
   client-selection prompt. (Richer templates such as meeting or QBR captures that
   have a =Client:= field additionally auto-fill the client display name; the action
   template has no such field.)
4. Finish with ~C-c C-c~ to file it (which also clocks out), or ~C-c C-k~ to
   abort.

Open =acme.org= and confirm the new =TODO= sits under =** Actions=. That round
trip — pick a key, answer the prompts, it lands in the right section already
tagged — is the entire capture model. Two things make it work:

- *Routing.* Each template targets a section heading by name in the active
  client's hub. No inbox, no manual refiling.
- *Every template is a file you can override.* All captures are filled from a
  named =.org= file. At capture time the file resolves the active client's
  =templates/<name>.org= first, then the bundled template — so editing
  =acme/templates/standup.org= tailors the standup streams for Acme without
  touching any other client (see [[*Step 11 — Make it yours][Step 11]]). The standup is just one of these
  files; it has no special handling.

* Step 7 — A tour of every capture, in working order

All engagement captures live under ~C-c c e~. Here is the full set, grouped the
way you use them. For each: the key, what it captures, the hub section it files
into, and when to reach for it. (Field-by-field detail is in [[file:reference.org][reference.org]].)

** Action tracking & delegation

| Key  | Captures             | Files to       | Reach for it when…                          |
|------+----------------------+----------------+---------------------------------------------|
| =ew= | Action item (TODO)   | Actions        | A task lands on /your/ plate                |
| =eg= | Delegate (WAITING)   | Delegations    | You hand a task to someone — sets =WAITING= |
| =eb= | Blocker / escalation | Blockers       | Something is blocking delivery              |
| =eW= | Weekly review        | Weekly Reviews | Your Friday delegation + team pulse pass    |
| =eP= | Person / team note   | Person node    | Capacity, a concern, or a win to record     |

=eg= is the engine of the *Delegated* dashboard block: it files in the =WAITING=
state, which is how the system separates "things I owe" from "things I am owed."

*** Acting on an action that already exists

=ew=, =eg= and =eb= all *create* an item. Two commands instead act on the Org
heading already under your cursor, so you never retype work that is already
captured:

- =M-x org-fractional-cto-delegate-at-point= — hand off the action at point. It
  flips the heading to =WAITING=, adds the =DELEGATED= tag, records who now owns
  it (=ASSIGNED_TO=) and when (=DELEGATED_ON=), and schedules your follow-up
  check-in (=SCHEDULED=), optionally with a delivery =DEADLINE=. It is =eg=
  applied in place — the action moves from *Open actions* to *Delegated —
  awaiting response* on the next dashboard refresh.
- =M-x org-fractional-cto-block-at-point= — raise a blocker against the action
  at point. It files a =[#A]= blocker into the client's *Blockers* section with
  its =BLOCKING= property linked back to the action, and drops a back-reference
  under the action itself, so the two sides point at each other.

Both are bound to =g= and =b= in =org-fractional-cto-command-map= (see
[[*Step 10 — The management commands][Step 10]]). They can also be invoked
*directly on an agenda line*:

1. Open the dashboard (~C-c a E~) and move point to a line for an action you
   want to delegate.
2. Press =, g= (Evil motion state) — or your configured non-Evil prefix followed
   by =g=, or =M-x org-fractional-cto-delegate-at-point=.
3. You are prompted for the assignee, a check-in date (=SCHEDULED=), and an
   optional delivery deadline. The entry flips to =WAITING= / =DELEGATED= in the
   source hub and moves from *Open actions* to *Delegated — awaiting response* on
   the next agenda refresh (=r=).

For a blocker: press =, b= (or prefix =b= / =M-x org-fractional-cto-block-at-point=)
on an agenda line. A =[#A]= blocker is filed into the client's *Blockers* section
with a =BLOCKING= back-reference to the action, and the action gains a reciprocal
link. The blocker appears in the *Blockers & escalations* block immediately after
refresh.

Reach for these agenda-line shortcuts when the work is already captured; reach
for =eg= / =eb= when you are capturing something new.

* Step 8 — Read your dashboard

Press ~C-c a E~ (or ~M-x org-fractional-cto-dashboard~). You get a *global* board
spanning all client files, with nine blocks, in this order:

1. *Next 7 days* — scheduled items and deadlines from the hub.
2. *Delegated — awaiting response* — everything =WAITING=, deadline-sorted.
   This is your "work I am owed" inbox; nothing should sit here a full week
   without a check-in.
3. *Blockers & escalations* — anything tagged =BLOCKER=.
4. *Open actions* — your =TODO= / =NEXT= / =INPROGRESS= work, with blockers and
   commitments filtered out so they are not double-listed.
5. *Commitments* — promises made, by you or the client.
6. *Active risks* — the risk register (the section container is skipped, so you
   see the risks themselves, not the empty parent heading). A risk closed out
   with =Status: Resolved= or =Mitigated= drops off the board; =Open= and
   =Accepted= risks stay.
7. *Open security findings* — =[SECURITY]= findings, with closed-out ones
   (=Status: Resolved= or =Mitigated=) filtered out — the *same closed-set rule*
   as risks.
8. *Open tech debt* — every =[TECH DEBT]= item recorded for the client.
9. *Scope changes — pending decisions* — =SCOPE CHANGE= entries sorted by their
   "decision needed by" deadline, so a decision due beyond the seven-day agenda
   window is still visible here.

Blocks 6–9 are the *watch list*: severity-rated liabilities (risks, security
findings, tech debt) and scope decisions that carry no =TODO= keyword and so
would otherwise never appear in an action view.

*** Closing out a risk or security finding

To remove a risk or security finding from the board, jump to its heading (press
=RET= or =TAB= on the agenda line to visit the source entry) and change its
=Status:= line to =Resolved= or =Mitigated=. On the next dashboard refresh (=r=),
the entry leaves the *Active risks* or *Open security findings* block. Entries
with =Status: Open= or =Status: Accepted= remain visible.

*** Filtering the dashboard to one client

Because the dashboard is a *real Org agenda command*, not a bespoke function, it
inherits the dispatcher's powers: refresh with =r=, filter with =/=, narrow by
tag, export, and it is sticky-agenda aware. The dashboard spans *all* client
files. When an active client is set, the view opens pre-focused on that client
via the native =org-agenda-tag-filter-preset=, and the =CATEGORY= column shows
each entry's client display name. For example, with =acme= active:

- The agenda header line shows =[+ACME]= and every entry's =CATEGORY= column
  reads =Acme Corp=.
- To widen to all clients: press =/=, clear the filter input, and press =RET=.
- To refocus on a different client (e.g. Beta Inc with slug =beta=): press =/=,
  type =+BETA=, then press =RET=. The board narrows to Beta's entries.

=org-fractional-cto-set-tag-inheritance= (on by default) ensures the inherited
client filetag is visible to the filter.
=M-x org-fractional-cto-switch-client= sets a different client active and
re-opens the board pre-focused to it in one step.

* Step 9 — Look across all clients

With an active client the dashboard opens focused; clear the filter (=/= then
=<RET>=) to go portfolio-wide without leaving the board. For raw cross-client
lookups by type, use the plain agenda dispatcher:

| Keys                | Shows                                   |
|---------------------+-----------------------------------------|
| ~C-c a m RISK~      | Every risk, across all clients          |
| ~C-c a m DELEGATED~ | Everything you have delegated, anywhere |
| ~C-c a m BLOCKER~   | All blockers everywhere                 |
| ~C-c a t~           | All TODOs across clients                |

The full tag legend (one tag per capture type) is in the README and
[[file:reference.org][reference.org]]. The pattern: =C-c a m <TAG>= turns any capture type into a
cross-client list.

* Step 10 — The management commands

The management commands live in =org-fractional-cto-command-map=. They are
available via =M-x= and also bound under the keys shown:

| Map key | Command                                  | Action                                           |
|---------+------------------------------------------+--------------------------------------------------|
| =n=     | =org-fractional-cto-new-client=          | New client (onboard)                             |
| =s=     | =org-fractional-cto-set-active-client=   | Set active client                                |
| =k=     | =org-fractional-cto-clear-active-client= | Clear active client                              |
| =d=     | =org-fractional-cto-dashboard=           | Global dashboard (pre-filtered to active client) |
| =w=     | =org-fractional-cto-switch-client=       | Switch client + open dashboard                   |
| =p=     | =org-fractional-cto-new-prospect=        | New prospect (LEAD stage, sets active)           |
| =S=     | =org-fractional-cto-set-stage=           | Set engagement stage on active client's hub      |
| =g=     | =org-fractional-cto-delegate-at-point=   | Delegate the action at point                     |
| =b=     | =org-fractional-cto-block-at-point=      | Blocker against action at point                  |
| =h=     | =org-fractional-cto-docs=                | Open the manuals                                 |

=g= and =b= act on the Org heading at point — both *from within a hub file* and
*directly from an agenda line* (see [[*Acting on an action that already exists][Step 7]]); the rest are session-level management
commands.

The map is *not* bound to any prefix by default (=org-fractional-cto-keymap-prefix=
is =nil=), because common prefixes like =C-c f= are already taken (Magit). Opt
in by setting the prefix *before* =setup=:

#+begin_src emacs-lisp
(setq org-fractional-cto-keymap-prefix "C-c k")  ; then n/s/k/d/w under C-c k
#+end_src

Two customisations you are most likely to want:

** Override a template for one client

Each client has its own =templates/= folder, populated at onboarding with a copy
of every bundled capture template. To customise a template for that client, edit
the file under =<slug>/templates/= — for example, rename the standup streams in
=acme/templates/standup.org= to the real Acme team.

At capture time a template resolves in two steps: (1) the active client's
=templates/<name>.org= if it exists, else (2) the bundled template
(=org-fractional-cto-template-directory=, default = the package's =templates/=).
There is no flat =<slug>/standup.org= fallback; per-client overrides live only
under =templates/=.

The =templates/= folder is *not* scanned by the agenda — Org lists agenda
directories non-recursively and only the client root dirs are added to
=org-agenda-files=, so per-client template files never appear on the dashboard.

If a capture template clocks in, keep its =%?= cursor marker in the *top entry
body* — above the first =**= sub-heading — so Org clocks the entry's main
heading rather than a sub-heading. Preserve that placement when you edit a
per-client clock-in template.

*Existing clients.* Clients created before this feature have no =templates/=
folder; they keep working via the bundled templates — nothing breaks.

To adopt per-client overrides for an existing client, create the =templates/=
directory under that client and copy in the bundled templates you want to
customise. The quickest way to copy them all is to evaluate, in a scratch
buffer, ~(org-fractional-cto--copy-templates "<slug>")~ — this copies the
bundled set into =<slug>/templates/=, skipping any files already there (it never
touches your hub or =CONTEXT.md=). Alternatively, copy the bundled
=templates/*.org= files in by hand.

If you had previously edited a flat =<slug>/standup.org=, move it to
=<slug>/templates/standup.org=.

*Do not re-run =org-fractional-cto-new-client= on an existing client to get
templates.* =new-client= rewrites the client's hub (=<slug>.org=, which holds all
your captured notes) and its =CONTEXT.md=, so running it on an active client
would overwrite that data — it is guarded only by a yes/no prompt. Use the
=org-fractional-cto--copy-templates= path above, which only ever adds missing
template files.

** Override every template globally

This sets the fallback templates used for *all* clients that have no matching
per-client =templates/<name>.org= override.

Copy the bundled =templates/= folder somewhere of your own, edit only the file
you care about (say =adr.org=), and point the variable at your copy:

#+begin_src emacs-lisp
(setq org-fractional-cto-template-directory "~/org/ofc-templates")
#+end_src

This becomes the fallback for clients with no matching =templates/<name>.org=
override.

** Reorder or extend the dashboard

=org-fractional-cto-dashboard-blocks= is an ordinary list of Org agenda series
entries. Drop a block, reorder them, or add your own — for example, a block for
=NEXT= items first — then re-run ~M-x org-fractional-cto-setup~ to apply it (the
install is idempotent, so this refreshes the command in place).

* Step 12 — Wire in AI session skills (optional)

If you also do hands-on implementation with an AI agent, two skill packs pair
well with the workflow. They live in your *code* repositories, not here; the
link between them and Org is the =CONTEXT.md= file plus the captured artifacts.
The loop is: Org tells you /what/ and /why/ → the skill helps you /do/ it → you
capture the /outcome/ back into Org. See the README's
"Working with AI session skills" section for the per-capture skill mapping.

* Step 12b — Let AI draft your action items (optional)

When you finalize a standup, meeting, or retrospective capture,
=org-fractional-cto= can pass the note's prose to a language-model backend and
surface candidate *Actions*, *Risks*, *Blockers*, and *Decisions* for your
review before anything is filed. The model proposes; you decide — nothing lands
in your hub without a deliberate ~C-c C-c~.

The feature is *off by default*. Until you set
=org-fractional-cto-ai-request-function=, captures are completely unaffected.

** 1. Wire up a backend

Set =org-fractional-cto-ai-request-function= to any function with the signature
=(PROMPT CALLBACK)=: your function calls CALLBACK with the model's raw response
string (or =nil= on failure). The call may be synchronous or async; the engine
always defers it so finalize never blocks and a backend failure never aborts
the capture.

The README's "AI note extraction" section has a ready-to-paste backend wired to
the [[https://llm.datasette.io/][llm]] CLI (any CLI reading a prompt on stdin works equally well):

#+begin_src emacs-lisp
(setq org-fractional-cto-ai-request-function
      (lambda (prompt callback)
        ;; … see README for the full make-process implementation …
        ))
#+end_src

Any HTTP client or Emacs-native LLM library works just as well — the package
only cares about the =(PROMPT CALLBACK)= contract.

** 2. Capture and finalize a standup

Press ~C-c c e s~ to open a standup capture. Fill in yesterday, today, and any
blockers, then press ~C-c C-c~ to finalize.

The capture files immediately — you are back at your buffer in the usual time.
The model call runs in the background; you do not wait for it.

** 3. Curate the review buffer

When the model responds, the =*ofc-ai-review*= buffer pops up. It shows each
extracted item as an Org heading in the same shape it will take in the hub —
for example:

: ** TODO Follow up with design team on mockup sign-off
: ** [RISK] Delayed API access may push the sprint boundary       :RISK:
: ** TODO [#A] BLOCKER: No staging environment blocks integration :BLOCKER:

Each heading also carries an =:OFC_AI_SECTION:= property naming the hub section
it will be filed under (and an =:OFC_AI_OWNER:= property when the model named an
owner). Edit freely: fix wording, change priority, adjust the owner. Delete any
subtree you do not want to keep. Normal Org editing commands all work; the
buffer stays open until you act on it, and other captures continue unimpeded.
The =:AI:= tag and =Source:= back-link are added when you file, not here.

** 4. File the keepers

- ~C-c C-c~ — files every surviving entry into the hub under the appropriate
  section (=Actions=, =Risks=, =Blockers=, or =Decisions=).
- ~C-c C-k~ — discards all candidates without filing anything.

Each filed item receives two provenance annotations automatically:

- =Source: [[id:…][<note-title>]]= — an Org link back to the standup note.
- The =:AI:= tag (configurable via =org-fractional-cto-ai-provenance-tag=; set
  it to =nil= to suppress tagging entirely).

An action filed this way lands under =** Actions= just like a hand-captured
=ew= item, with the same =TODO= keyword and client tag — it will appear on your
dashboard the next time you press ~C-c a E~ or refresh with =r=.

** Which captures are flagged by default

Seven templates carry =:ofc-ai-extract t= out of the box:

| Key   | Capture           |
|-------+-------------------|
| =es=  | Standup           |
| =ed=  | Discovery session |
| =em=  | Client meeting    |
| =ei=  | Internal sync     |
| =eW=  | Weekly review     |
| =eq=  | QBR               |
| =eR=  | Retrospective     |

All other captures are unaffected. To opt in a custom template, add
=:ofc-ai-extract t= to its =org-capture-templates= plist entry.

** A note on duplicates

Re-finalizing the same note re-runs extraction and produces duplicate
candidates. Duplicate detection is not performed in v1 — review the buffer
carefully and discard any items you have already filed.

* Step 13 — Settle into the rhythm

You now know every moving part. The operating cadence ties them together:

- *Daily* — attend standups (=es=), capture blockers (=eb=) as they surface.
- *Weekly* — delegation + pulse review (=eW=); internal sync (=ei=) before any
  client comms; client sync (=em=); sprint retro (=eR=). Clear the *Delegated*
  block on the dashboard.
- *Monthly* — client health check (=eh=); review the innovation pipeline (=en=).
- *Quarterly* — QBR (=eq=); architecture health review (=eA=).

The /why/ behind this cadence — the engagement phases, the "hat" model, and the
decision frameworks — is the job of [[file:playbook.org][playbook.org]]. Read it next.

* Troubleshooting

- *Every capture prompts me for a client.* No active client is set. Run
  =org-fractional-cto-set-active-client= (or =switch-client=). This is expected
  behaviour, not a bug — it just means one extra keystroke per capture.

- *An =INPROGRESS= task is missing from the dashboard's Open actions.* First
  confirm =org-fractional-cto-setup= ran — it registers =INPROGRESS= globally, so
  if it never ran (and the file lacks a =#+TODO= line) the state is unknown.
  Second, the keyword must be exactly =INPROGRESS=, with *no hyphen*. Org's
  agenda match language treats =-= as a boolean NOT, so a hyphenated keyword
  (=IN-PROGRESS=) is parsed as "=IN= and not =PROGRESS=" and never matches.
  Scaffolded hubs use the hyphen-free form; if you have an old hub, fix its
  =#+TODO= line and any task states.

- *The dashboard shows no entries for my client.* The dashboard now spans all
  client files. If no active client is set, all clients appear unfiltered. Set
  one with =M-x org-fractional-cto-set-active-client= to focus the board. If the
  filter is not working, ensure =org-fractional-cto-set-tag-inheritance= is =t= (the
  default) so the inherited client filetag is visible to the agenda filter.

- *A standup capture inserts literal =%^{...}= / =%U= text.* You are on an old
  version where the standup was injected as a string. Update the package; every
  template (the standup included) is now read as a file so its escapes expand
  normally.

- *A clocked capture clocks a sub-heading instead of the entry.* The template's
  =%?= cursor marker sits below the first =**= sub-heading. Move =%?= into the
  top entry body, above the first sub-heading, so Org clocks the main heading.

- *~C-c a E~ is not listed in the dispatcher.* =setup= has not run (or ran
  before Org loaded). Ensure =org-fractional-cto-setup= runs =:after org=, then
  re-run it — it is idempotent.

- *I changed config but nothing updated.* Re-run ~M-x org-fractional-cto-setup~.
  It refreshes captures and the dashboard in place; no restart needed.

- *Filtering by client shows other clients' entries (or nothing).* The dashboard
  filter relies on the inherited =#+filetags= client tag being visible to the Org
  agenda. Ensure =org-fractional-cto-set-tag-inheritance= is =t= (the default) and
  that =org-fractional-cto-setup= has run — it sets =org-agenda-use-tag-inheritance=
  to =t=. If you set the variable to =nil= to manage tag inheritance yourself, you
  must enable it globally or the filter will silently match nothing.

- *The agenda at-point prefix does nothing.* Set
  =org-fractional-cto-agenda-keymap-prefix= to your chosen prefix key *before*
  calling =org-fractional-cto-setup=, then re-run ~M-x org-fractional-cto-setup~.
  Also verify the chosen prefix does not shadow an existing agenda binding (e.g.
  =s= is already bound in =org-agenda-mode-map= as =org-agenda-schedule=).  If you
  use Evil, the comma localleader (=, g= / =, b=) is wired automatically by setup
  and does not require a prefix.

* People & Stakeholders

People are first-class Org nodes, not inline text.  Each person is one file
under =org-fractional-cto-people-directory= (default =~/org/people=), a
file-level =org-id= node with an =:ID:= and a =#+title=.  This record is
global and durable: it survives any client being archived or deleted, and is
where long-term, cross-engagement history about a person accumulates.

A *stakeholder profile* is different: it lives in the client hub's
=Stakeholder Profiles= section and captures engagement-specific relationship
data (influence, what success means to them, communication cadence).  It links
to the global person via its =Person (global node):= line.  One person can be a
stakeholder at several clients.

** Referencing a person

- =M-x org-fractional-cto-insert-person= — complete by name and insert an
  =[[id:...][Name]]= link; type a new name to create the node on the fly.
- Or the native dance: visit the person node, =C-c l= (=org-store-link=), then
  =C-c C-l= (=org-insert-link=) where you are writing.
- The =eP= capture (=C-c c e P=) files a timestamped note under a person's
  =Notes / History=, creating the node if needed.

** org-roam users

org-roam is *not* required.  If you already use roam, set
=org-fractional-cto-people-directory= to a folder inside your
=org-roam-directory=; roam then indexes these nodes and your
=org-roam-node-insert= and backlinks buffer work unchanged.

** Assigning and referencing people

Assignment and reference fields link to person nodes instead of storing bare
names.  When a capture or command asks for an owner, assignee, attendee, or
author, type a name: an existing person is completed and linked, a new name
creates the node on the spot.  The field stores an =[[id:...][Name]]= link.

Five single-owner item types also tag the heading with the person's =@slug=
tag — delegation, blocker, commitment, risk, and security.  Because they are
tags, the dashboard slices by person natively: open the dashboard, press =/=
and type the tag (e.g. =@jane_doe=) to see just that person's open work.  This
needs no extra command and works without org-roam.

=M-x org-fractional-cto-delegate-at-point= and
=M-x org-fractional-cto-block-at-point= do the same for an existing heading:
they link the assignee / unblock-owner and add the =@slug= tag.

Owner *columns inside tables* are filled by hand — use
=M-x org-fractional-cto-insert-person= in the cell.

Existing notes with plain-text owners keep working; re-assigning through the
capture or command upgrades them to a link and tag in place.
