#+title: org-fractional-cto
#+subtitle: An opinionated Org-mode workflow for running fractional-CTO engagements
#+options: toc:2 num:nil

[[https://www.gnu.org/licenses/gpl-3.0][https://img.shields.io/badge/license-GPLv3-blue.svg]]

* What it is

=org-fractional-cto= turns Org mode into an operating system for running
technical client engagements as a *fractional CTO* or *engagement director*.

It is opinionated: it encodes a particular way of working — onboard a client,
discover the system, make and record decisions, run a delivery cadence,
delegate and track work, manage people, and grow the account — and gives you
the Org scaffolding to do all of it without leaving Emacs.

Three moving parts, one workflow:

1. *Onboarding* — =org-fractional-cto-new-client= scaffolds a complete
   per-client workspace in seconds (operational hub, a =templates/= folder of
   per-client capture templates, and a context file), or
   =org-fractional-cto-new-prospect= does the same for a pre-sales lead.
2. *Capture* — a single =e= prefix (~C-c c e ...~) routes every kind of note —
   pre-sales calls, research, discovery, ADRs, risks, commitments, delegated
   tasks, blockers, retrospectives, QBRs, innovation ideas — straight into the
   active client's file. No per-client key prefixes.
3. *Dashboard* — a global agenda view (~C-c a E~) surfacing delegated work,
   blockers, open actions, commitments, and active risks across all clients.
   When an active client is set, it opens pre-focused on that client.

Every engagement also carries a *stage* — from =LEAD= (a fresh pre-sales
prospect) through =QUALIFIED= and =ACTIVE= to =LOST= / =DORMANT= — and a
cross-client *pipeline* view (~C-c a P~) lists your whole funnel, one line per
prospect.

It is *multi-client by design*. Set an "active client" once per session and
every capture and the dashboard follow it. Switch clients with one command.

* Why it is built this way

- *One =e= prefix, not one prefix per client.* Adding a client never means
  editing capture config. The active-client variable does the routing.
- *The dashboard is a real Org agenda command* (registered under ~C-c a E~), not
  a bespoke function. It therefore inherits everything the agenda dispatcher
  gives you: it appears in the ~C-c a~ menu, it is exportable, filterable with
  =/=, refreshable with =r=, and sticky-agenda aware. It spans *all* client files;
  when an active client is set it opens pre-focused via the native
  =org-agenda-tag-filter-preset=. The =CATEGORY= column shows each entry's client.
- *Captures file directly into the client hub*, not into a refile inbox. Open
  =<client>.org= and the whole engagement is there, organised by section.
- *Templates are per-client and overridable.* Every capture template ships with
  the package and is copied into each client's =templates/= folder at
  onboarding. A capture uses the client's =templates/<name>.org= if present, else
  the bundled template — so edit one client's copy to tailor a capture for that
  engagement, or point =org-fractional-cto-template-directory= at your own folder
  to change the defaults for every client.

* Installation

Requires Emacs 27.1+ and Org 9.4+.

** use-package

*** From a local clone
#+begin_src emacs-lisp
(use-package org-fractional-cto
  :load-path "~/src/org-fractional-cto"
  :after org
  :custom
  (org-fractional-cto-clients-directory "~/org/clients")
  (org-fractional-cto-author "Your Name")
  :config
  (org-fractional-cto-setup))
#+end_src
*** Using straight
#+begin_src emacs-lisp
(use-package org-fractional-cto
  :after org
  :straight (:type git :host github :repo "dhruvasagar/org-fractional-cto" :files (:defaults "templates" "doc"))
  :custom
  (org-fractional-cto-clients-directory "~/org/clients")
  (org-fractional-cto-author "Your Name")
  :config
  (org-fractional-cto-setup))
#+end_src

** Manual

#+begin_src emacs-lisp
(add-to-list 'load-path "~/src/org-fractional-cto")
(require 'org-fractional-cto)
(setq org-fractional-cto-clients-directory "~/org/clients")
(org-fractional-cto-setup)

=org-fractional-cto-setup= installs the captures, registers the dashboard, and
adds your client directories to =org-agenda-files= (see [[*Key bindings & legend][Key bindings]]).  It binds
the management command map only if you set =org-fractional-cto-keymap-prefix=
(nil by default, so no global key is claimed).

* Core concepts

** The active client

Most of the workflow is scoped to one client at a time.

| Command                                      | Effect                                |
|----------------------------------------------+---------------------------------------|
| =M-x org-fractional-cto-set-active-client=   | Pick the client for this session      |
| =M-x org-fractional-cto-clear-active-client= | Go back to being prompted per capture |
| =M-x org-fractional-cto-switch-client=       | Set active *and* open its dashboard   |

With an active client set, ~C-c c e ...~ and ~C-c a E~ never prompt. With none
set, they prompt you to choose — nothing breaks, it is just one extra step.

** The client hub

Each client owns a directory:

#+begin_example
<clients-directory>/
  acme/
    acme.org      ← operational hub: one section per capture type
    CONTEXT.md    ← domain glossary, people, tech stack (load in AI sessions)
    templates/    ← per-client copies of every capture template (override-able)
#+end_example

Onboarding copies the full bundled set of capture templates into =templates/=.
At capture time a template resolves to the active client's
=<slug>/templates/<name>.org= if it exists, else the bundled template — so edit
a file under a client's =templates/= to tailor it for that client. The folder is
not scanned by the agenda. See [[*Customisation][Customisation]] for details.

The operational hub declares the client identity once as a file-level tag, so it
is inherited by every heading in the file rather than repeated. Onboarding
generates the whole file — header, engagement heading, and one empty section per
capture type, each pre-tagged with its type subtag:

#+begin_example
#+title: Acme Corp
#+AUTHOR: Your Name
#+STARTUP: overview
#+TODO: TODO NEXT INPROGRESS WAITING | DONE CANCELLED
#+filetags: :ACME:
#+OPTIONS: date:nil

* Acme Corp Engagement  :ACTIVE:
:PROPERTIES:
:ID:       ACME-OPS
:CATEGORY: Acme Corp
:END:

See [[file:CONTEXT.md][CONTEXT.md]] for domain vocabulary, key people, and priorities.

** Actions
** Delegations  :DELEGATED:
** Blockers  :BLOCKER:
** People  :PEOPLE:
** Weekly Reviews  :REVIEW:
** Commitments  :COMMITMENT:
** Risks  :RISK:
** Meeting Notes  :MEETING:
** Standup Notes  :STANDUP:
** Internal Syncs  :INTERNAL:
** Discovery Sessions  :DISCOVERY:
** Stakeholder Profiles  :STAKEHOLDER:
** Architecture Decisions  :ADR:
** Architecture Reviews  :ARCHREVIEW:
** Vendor Evaluations  :VENDOR:
** Technical Debt  :TECHDEBT:
** Security Findings  :SECURITY:
** Health Checks  :HEALTH:
** QBRs  :QBR:
** Innovation Pipeline  :INNOVATION:
** Scope Changes  :SCOPE:
** Post-Mortems  :POSTMORTEM:
** Retrospectives  :RETRO:
** Pre-Sales Notes  :PRESALES:
** Research  :RESEARCH:
** Qualification  :QUALIFICATION:
#+end_example

The =* … Engagement= heading carries the *stage* tag (=:ACTIVE:= here) and an
=:ID:= / =:CATEGORY:= drawer — the =CATEGORY= is what the dashboard's category
column shows. Every =**= section is created empty, ready to receive captures.
The client tag (=:ACME:=) is never repeated per-heading; Org's tag inheritance
propagates it from =#+filetags= to every section and captured item.
Existing hubs can be migrated to this layout with =M-x org-fractional-cto-upgrade-hub=.

The hub's sections, the capture targets, and the dashboard all agree by
construction — they are generated from one section list.

** People & stakeholders

People are first-class Org nodes, not inline text.  Each person is one file
under =org-fractional-cto-people-directory= (default =~/org/people=, a sibling
of the clients directory), holding a file-level =org-id= node with an =:ID:=
property and a =#+title=.  This record is *global and durable*: it survives any
client being archived or deleted, and is where long-term, cross-engagement
history about a person accumulates.

A *stakeholder profile* is different: it lives in the client hub's
=Stakeholder Profiles= section and captures engagement-specific relationship
data.  It links to the global person via its =Person (global node):= line.
One person can be a stakeholder at several clients.

Assignment and reference fields (assignee, owner, attendees, author) link to
person nodes via =[[id:...][Name]]=.  Type a name at any prompt: an existing
person is completed and linked; a new name creates the node on the spot.  For
owner columns inside tables, use =M-x org-fractional-cto-insert-person= in the
cell.

Five single-owner item types — delegation, blocker, commitment, risk, security
— also tag the heading with the person's =@slug= (e.g. =@jane_doe=).  Press
=/= in the dashboard and type the tag to filter to that person's open work.

To insert a standalone person link: =M-x org-fractional-cto-insert-person=.
The =eP= capture (=C-c c e P=) files a timestamped note under a person's
=Notes / History=, creating the node if needed.

org-roam is *not* required.  If you use roam, set
=org-fractional-cto-people-directory= to a folder inside your
=org-roam-directory= and roam indexes the nodes automatically.

* Quick start

1. =M-x org-fractional-cto-new-client= → name = "Acme Corp", slug = "acme".
2. Fill in the =CONTEXT.md= that opens.
3. =M-x org-fractional-cto-set-active-client= → =acme=.
4. Capture: ~C-c c e w~ (an action), ~C-c c e g~ (delegate a task), ~C-c c e d~
   (a discovery session)…
5. Review: ~C-c a E~ opens the global dashboard, pre-focused on Acme.

Prospecting rather than onboarding a won client? Use =M-x
org-fractional-cto-new-prospect= in step 1 — it builds the same hub at the
=LEAD= stage and drops you straight into the pre-sales call capture (=el=). Then
~C-c a P~ shows your whole pipeline across clients.

New here and want the long way round? [[file:doc/guide.org][doc/guide.org]] is a step-by-step
walkthrough of every feature, in order, with "what you should see" at each step.

* Capture reference

All engagement captures are under ~C-c c e~. They file directly into the active
client's hub. When an active client is set, captures route there automatically —
no client-selection prompt. Templates that carry a client display-name field
(such as the meeting and QBR templates) auto-fill it from the active client; the
simple action template (=ew=) has no such field and does not embed the name.

A *post-mortem* (=ef=) is reactive — you write it because something failed. A
*retrospective* (=eR=) is routine — you run it every sprint whether or not
anything broke. Keep them separate.

The =er= (Risk) and =ex= (Security finding) templates include a
=Status: Open | Mitigated | Resolved | Accepted= field. Setting an item to
=Resolved= or =Mitigated= removes it from the dashboard's "Active risks" and
"Open security findings" blocks respectively — that is the standard way to close
one out without deleting the record.

* AI note extraction (optional)

When a flagged capture is finalized, =org-fractional-cto= can send the note
text to a language-model backend and surface the extracted items for review
before filing them.  The feature extracts four types of items from the prose:
*Actions*, *Risks*, *Blockers*, and *Decisions*.  They appear in an Org-native
review buffer where you edit or delete freely, then file the survivors into the
right hub sections with a single key, each carrying a =Source:= back-link to the
origin note.

The feature is *off by default*.  Nothing happens until
=org-fractional-cto-ai-request-function= is set.

** Wiring a backend

Set =org-fractional-cto-ai-request-function= to any function with the signature
=(PROMPT CALLBACK)=: your function calls CALLBACK with the model's raw response
string (or nil on failure).  The implementation may be synchronous or
asynchronous; the engine always defers the call so finalize never blocks.

A minimal backend using the [[https://llm.datasette.io/][llm]] CLI (any CLI that reads a prompt on stdin works):

#+begin_src emacs-lisp
(setq org-fractional-cto-ai-request-function
      (lambda (prompt callback)
        (let ((buf (generate-new-buffer " *ofc-ai*")))
          (make-process
           :name "ofc-ai" :buffer buf
           :command '("llm" "--no-stream")   ; any CLI reading the prompt on stdin
           :connection-type 'pipe
           :sentinel (lambda (proc _e)
                       (when (memq (process-status proc) '(exit signal))
                         (with-current-buffer (process-buffer proc)
                           (funcall callback (buffer-string)))
                         (kill-buffer (process-buffer proc)))))
          (process-send-string
           (get-buffer-process buf) prompt)
          (process-send-eof (get-buffer-process buf)))))
#+end_src

Any CLI, HTTP client, or Emacs-native LLM library can serve as the backend —
the package only cares about the =(PROMPT CALLBACK)= contract.

** The review flow

Once the backend is wired:

1. Finalize any flagged capture (=C-c C-c= in the capture buffer).
2. The model call runs asynchronously in the background.
3. When the model responds, the =*ofc-ai-review*= buffer pops up, showing each
   extracted item as an Org heading.  Edit the titles, delete subtrees you do
   not want to keep, or adjust any field.
4. Press ~C-c C-c~ to file every surviving entry into the hub, or ~C-c C-k~ to
   discard them all.

The buffer stays open until you act — normal editing and other captures
continue unimpeded.

** Filed items: provenance and tags

Each accepted entry is placed under the appropriate hub section (e.g. =Actions=,
=Risks=) at the correct heading depth and receives two provenance annotations:

- =Source: [[id:…][<note-title>]]=  — an Org link back to the originating note.
- The =:AI:= tag (configurable via =org-fractional-cto-ai-provenance-tag=; set
  it to =nil= to suppress tagging entirely).

** Configuration

| Variable                                       | Default   | Purpose                                           |
|------------------------------------------------+-----------+---------------------------------------------------|
| =org-fractional-cto-ai-request-function=       | =nil=     | Backend function =(PROMPT CALLBACK)=; nil disables |
| =org-fractional-cto-ai-item-types=             | (4 types) | Taxonomy: Actions, Risks, Blockers, Decisions     |
| =org-fractional-cto-ai-provenance-tag=         | ="AI"=    | Tag added to filed items; nil adds no tag         |

** Extending the taxonomy

=org-fractional-cto-ai-item-types= is an alist of =(TYPE . PLIST)= entries.  To
add a new extractable type, append a row:

#+begin_src emacs-lisp
(add-to-list 'org-fractional-cto-ai-item-types
             '(commitment
               :section "Commitments"
               :tag "COMMITMENT"
               :desc "A commitment or promise made by either party."
               :render my/ofc-ai-render-commitment)
             t)  ; append, not prepend
#+end_src

The =:section= value must match a heading name from =org-fractional-cto-sections=
(the hub's section list).  The =:render= function receives a normalized item
plist with keys =:type :title :owner :deadline :priority :body :fields= and must
return Org entry text.

** Which captures are flagged

The following seven capture templates carry =:ofc-ai-extract t= by default:

| Key  | Capture           |
|------+-------------------|
| =es= | Standup           |
| =ed= | Discovery session |
| =em= | Client meeting    |
| =ei= | Internal sync     |
| =eW= | Weekly review     |
| =eq= | QBR               |
| =eR= | Retrospective     |

To flag any other capture template, add =:ofc-ai-extract t= to its
=org-capture-templates= plist entry.  Per-client template overrides (files
under =<slug>/templates/=) are used for rendering, but the =:ofc-ai-extract=
flag is read from the template's Lisp-level entry, not the template file itself.

* The dashboard

~C-c a E~ (or =M-x org-fractional-cto-dashboard=) opens the *global* dashboard
spanning all client files. The 9 blocks are:

1. *Next 7 days* — scheduled items and deadlines.
2. *Delegated — awaiting response* — everything in the =WAITING= state, sorted
   by deadline. This is "work I am owed."
3. *Blockers & escalations* — anything tagged =BLOCKER=.
4. *Open actions* — your =TODO=/=NEXT=/=INPROGRESS= work, with blockers and
   commitments filtered out so they are not double-listed.
5. *Commitments* — promises made, by you or the client.
6. *Active risks* — the risk register. Risks closed out (=Status: Resolved= or
   =Mitigated=) drop off, so the block stays current instead of accumulating;
   =Open= and =Accepted= risks stay visible.
7. *Open security findings* — =[SECURITY]= findings, with closed-out ones
   (=Status: Resolved= or =Mitigated=) filtered out by the *same rule as risks*,
   priority- then deadline-sorted.
8. *Open tech debt* — every =[TECH DEBT]= item logged against the client.
9. *Scope changes — pending decisions* — =SCOPE CHANGE= entries, soonest
   "decision needed by" deadline first, so a decision due next month is visible
   now rather than only inside the seven-day window.

Blocks 6–9 are the engagement's *watch list*: liabilities that carry no =TODO=
keyword and would otherwise never surface in an action view. Reorder or extend
the blocks with =org-fractional-cto-dashboard-blocks=.

When an active client is set, the dashboard opens pre-focused on that client via
the native tag filter; the =CATEGORY= column shows which client each entry belongs to.
To change focus while the dashboard is open:
- Press =/= then type a tag expression such as =+ACME= to narrow to that client,
  or =-ACME= to exclude it.
- Press =/= then =RET= (empty input) to clear the filter and see all clients.
=M-x org-fractional-cto-switch-client= sets a different client active
and opens the board in one step — the fast path for context switching.

For the cross-client view, ~C-c a P~ (=M-x org-fractional-cto-pipeline=) lists
every =LEAD= / =QUALIFIED= engagement across all clients, one line per prospect,
grouped by stage (all =LEAD= entries before =QUALIFIED=, then by client) — your
funnel board, complementing the global =C-c a E= dashboard. The header tracks
=org-fractional-cto-pipeline-stages=, so it stays accurate if you customise the
stages shown.

** Key bindings

*** At-point actions (from the agenda or a hub file)

=M-x org-fractional-cto-delegate-at-point= and
=M-x org-fractional-cto-block-at-point= can be called directly on any
Org agenda line (or from a hub heading):

Both commands work in the hub buffer AND from an agenda line.

- *Evil users* — the comma localleader binds them in any Org agenda
  buffer: =, g= (delegate) and =, b= (block) in motion state.
- *Non-Evil users* — set =org-fractional-cto-agenda-keymap-prefix= to a
  key prefix (e.g. ="C-c C-'"=) to access them via a prefix key;
  otherwise use =M-x=.

*delegate* flips the heading to =WAITING=, tags it =DELEGATED=, records an
=ASSIGNED_TO= property and a delegated-on timestamp, and schedules a follow-up
check-in (with an optional delivery deadline). Prompts for assignee (required),
check-in date, and an optional deadline.

*block* files a =[#A] BLOCKER= item into the client's Blockers section with a
=BLOCKING= property linking back to the originating action, and inserts a
back-reference under that action. Prompts for what is blocked, who can clear it,
and an optional resolve-by date.

|-------------------------------------------+--------------------------------------------------|
| =org-fractional-cto-new-client=           | New client (onboard, ACTIVE)                     |
| =org-fractional-cto-new-prospect=         | New prospect (LEAD stage)                        |
| =org-fractional-cto-set-active-client=    | Set active client                                |
| =org-fractional-cto-clear-active-client=  | Clear active client                              |
| =org-fractional-cto-set-stage=            | Set the engagement stage                         |
| =org-fractional-cto-dashboard=            | Global dashboard (pre-filtered to active client) |
| =org-fractional-cto-switch-client=        | Switch client + open dashboard                   |
| =org-fractional-cto-delegate-at-point=    | Delegate the heading at point                    |
| =org-fractional-cto-block-at-point=       | Blocker against heading at point                 |
| =org-fractional-cto-upgrade-hub=          | Upgrade an old hub in place      |
| =org-fractional-cto-docs=                 | Open the bundled manuals         |

Most also live in =org-fractional-cto-command-map= (keys =n p s k S d w g b h=;
=upgrade-hub= is =M-x=-only).  It is
*not* bound to any prefix by default — =org-fractional-cto-keymap-prefix= is

** TODO keywords

The workflow is self-contained: =org-fractional-cto-setup= registers the keyword
sequence it needs into the global =org-todo-keywords= (and a face for
=INPROGRESS=), so these states are recognised in *every* Org buffer and agenda
match — you do not have to touch your personal Org configuration. The merge is
additive and idempotent: a sequence is added only when it introduces a keyword
Org does not already know, and any face you have defined wins. Customize the
sequence with =org-fractional-cto-todo-keywords= (set it to nil to manage
keywords entirely yourself).

Scaffolded client files additionally declare the same line locally, so a hub
opened on a machine that never ran setup still cycles the right states:

#+begin_example
#+TODO: TODO NEXT INPROGRESS WAITING | DONE CANCELLED
#+end_example

| Keyword       | Meaning in this workflow                                 |
|---------------+----------------------------------------------------------|
| =TODO=        | Open action you own                                      |
| =NEXT=        | The next physical action (GTD-style)                     |
| =INPROGRESS=  | Actively being worked                                    |
| =WAITING=     | *Delegated* — owned by someone else, you are tracking it |
| =DONE=        | Completed                                                |
| =CANCELLED=   | Dropped                                                  |

=WAITING= is load-bearing: it is what separates "things I owe" from "things I
am owed," and it feeds the dashboard's *Delegated* block.

** Engagement stages

Distinct from the TODO keywords above, a single *stage* tag sits on the top
=* … Engagement= heading (and is inherited by the whole subtree). Switch it with
=M-x org-fractional-cto-set-stage=.

| Stage       | Meaning                                      |
|-------------+----------------------------------------------|
| =LEAD=      | Captured from a pre-sales call; raw          |
| =QUALIFIED= | Researched and worth pursuing (pre-contract) |
| =ACTIVE=    | Won / engaged — the default for new-client   |
| =LOST=      | Did not convert                              |
| =DORMANT=   | Paused / on hold                             |

=org-fractional-cto-new-prospect= starts an engagement at =LEAD=;
=org-fractional-cto-new-client= starts it at =ACTIVE=. The pipeline view
(~C-c a P~) lists every =LEAD= / =QUALIFIED= engagement across all clients.

** Tags

Every capture carries a *type tag*. The client identity is inherited from the
hub's =#+filetags= line (e.g. =#+filetags: :ACME:=), so the client tag is never
repeated on individual headings or captured items.

* New client onboarding

=M-x org-fractional-cto-new-client= asks for a display name and a slug, then
creates:

- =<slug>/<slug>.org= — the hub, with the client identity in =#+filetags= and every section pre-created with its type subtag.
- =<slug>/CONTEXT.md= — fill this with domain vocabulary, key people, and the
  tech stack. Load it at the start of any AI coding session for that client.
- =<slug>/templates/= — a per-client copy of every capture template. Edit any
  file here to tailor that template for the client (e.g. rename the standup
  streams in =<slug>/templates/standup.org=). See [[*Customisation][Customisation]].

The new client is added to =org-agenda-files= immediately — no restart.

*Prospects.* =M-x org-fractional-cto-new-prospect= builds the identical
workspace but stamps the engagement at the =LEAD= stage, sets it active, and
opens the pre-sales call capture. As a lead progresses, advance it with =M-x
org-fractional-cto-set-stage= (=LEAD= → =QUALIFIED= → =ACTIVE= on a win, or
=LOST= / =DORMANT=). Hubs created before filetags / stages were introduced can be migrated with
=M-x org-fractional-cto-upgrade-hub= — it adds =#+filetags= for the client tag,
strips per-heading client tags, adds any missing sections, ensures the
engagement heading carries a stage tag (defaulting to =ACTIVE=, or whatever
=org-fractional-cto-default-stage= is set to), and copies any bundled capture
templates the client's =templates/= directory is missing (edited overrides are
left untouched). The command is idempotent.

*Per-client template overrides.* Each client gets a =templates/= folder of
capture templates, copied in at onboarding. A capture uses the client's
=templates/<name>.org= if present, otherwise the bundled template. Edit a
client's file to tailor that capture; see the guide's "Override a template for
one client" for details.

To customise *every* client's fallback at once, copy the bundled =templates/=
folder somewhere, edit the file you care about, and point
=org-fractional-cto-template-directory= at your copy.

* Working with AI session skills (optional)

This package manages the *record* of an engagement. When you also do hands-on
implementation with an AI agent (Claude Code, etc.), two skill packs pair well
with it. They are entirely optional and live in your *code* repositories, not
here — the connection is the =CONTEXT.md= file plus the captured artifacts.

** How they associate with the current work item

There is no magic binding between an Org heading and a skill. The association
is deliberate and manual, and that is the point:

1. *Pick the work item* in Org — an ADR you are about to write (=ea=), a tech
   spike (=ek=), a discovery session (=ed=), a security review (=ex=).
2. *Open the client's code repo* in your AI tool and load its =CONTEXT.md= so
   the agent has the domain vocabulary.
3. *Run the matching skill* (below) in that repo's AI session.
4. *Capture the result* back into Org with the relevant =e= key.

So the flow is: Org tells you /what/ and /why/; the skill helps you /do/ it; Org
records the /outcome/.

** [[https://github.com/mattpocock/skills][mattpocock/skills]] — calibration before a session

Install in the client repo: =npx skills@latest add mattpocock/skills=, then run
=/setup-matt-pocock-skills= once.

| Before this capture | Run this skill     | Why                                                      |
|---------------------+--------------------+----------------------------------------------------------|
| Discovery (=ed=)    | =/grill-me=        | Surface your assumptions before you anchor the interview |
| Discovery on a doc  | =/grill-with-docs= | Same, anchored to an SOW/PRD                             |
| Any unfamiliar area | =/zoom-out=        | Map the system before touching it                        |
| Session handoff     | =/handoff=         | Compact context so the next session starts fresh         |

** [[https://github.com/garrytan/gstack][gstack]] — structure during execution

Install globally: =git clone https://github.com/garrytan/gstack ~/.claude/skills/gstack && ./setup=.

| Before / during this capture      | Run this skill     | Why                                             |
|-----------------------------------+--------------------+-------------------------------------------------|
| ADR / Quick decision (=ea=, =eD=) | =/office-hours=    | Six forcing questions to challenge your framing |
| Architecture review (=eA=)        | =/plan-eng-review= | Engineering review with diagrams                |
| Security finding (=ex=)           | =/cso=             | OWASP Top 10 + STRIDE threat model              |
| Retrospective (=eR=)              | =/retro=           | Facilitation prompt for the sprint retro        |
| After any implementation          | =/review=          | Staff-engineer audit                            |

*Rule of thumb:* run =/office-hours= (gstack) or =/grill-me= (mattpocock) at the
/start/ of any non-trivial session. The sessions where you feel you least need
them are usually the ones where you most do.

* The methodology in brief

The package encodes a role that operates across several modes ("hats"). The
discipline is not to switch hats mid-session.

| Hat                 | When                              | Captures            |
|---------------------+-----------------------------------+---------------------|
| Trusted Advisor     | Pre-sales calls, research, qualifying | =el eo eF=      |
| Engagement Director | Client calls, QBRs, scope         | =em ec eq eh ee=    |
| Discovery           | Stream interviews, system mapping | =ed ek=             |
| Fractional CTO      | ADRs, roadmap, build-vs-buy       | =ea eD eA ev=       |
| People Manager      | Delegation, standups, retros      | =eg eb eW eP es eR= |
| QA / Security       | Coverage, compliance posture      | =ex et=             |

A typical cadence:

- *Daily*: attend standups (=es=), capture blockers (=eb=) as they surface.
- *Weekly*: delegation + pulse review (=eW=); internal sync (=ei=) before any
  client-facing comms; client sync (=em=); sprint retro (=eR=).
- *Monthly*: client health check (=eh=); review the innovation pipeline (=en=).
- *Quarterly*: QBR (=eq=); architecture health review (=eA=).

This is the short version. Three bundled guides go deeper:

- [[file:doc/guide.org][doc/guide.org]] — the step-by-step *user guide*: install and verify, onboard a
  client, tour every capture, read the dashboard, and tailor the system, in the
  order you meet them. Start here if you want a walkthrough rather than a lookup.
- [[file:doc/playbook.org][doc/playbook.org]] — the engagement phases (setup → discovery → architecture →
  operating rhythm → growth), the hat model in depth, and decision frameworks
  (build-vs-buy, ADR triggers, scope discipline).
- [[file:doc/reference.org][doc/reference.org]] — per-capture guidance, the operating rhythm day-by-day, the
  tag/keyword legend, AI-skill integration, and scenario guides ("I just had a
  discovery session", "something went wrong in production", …).
